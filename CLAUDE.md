## 🎯 О проекте

**Aim** — нативное Android-приложение для трекинга целей и привычек. Один пользователь, локальное хранилище, без облака. Премиальный UX в стиле Bold/Bento.

## 🚦 Текущий спринт

> Обновляй этот блок при переходе к следующему спринту.

**Активный спринт:** Спринт 1 — Foundation  
**Статус:** каркас собран в репозитории, ждёт верификации сборкой на машине с Android SDK  
**Следующая задача:** прогнать `./gradlew assembleDebug` + `./gradlew test`, проверить рендер `@Preview` в светлой/тёмной теме, затем переход к Спринту 2 (Room, Цели, Задачи).

## 🛠️ Стек (краткая шпаргалка)

- **Язык:** Kotlin 2.0+
- **UI:** Jetpack Compose + Material 3
- **DI:** Hilt
- **БД:** Room
- **Async:** Coroutines + StateFlow
- **Settings:** DataStore (Preferences)
- **Расписания:** WorkManager + AlarmManager
- **Виджеты:** Jetpack Glance
- **Min SDK:** 33 (Android 13)
- **Target SDK:** 35 (Android 15)

## 🏛️ Архитектура (помни всегда)

```
presentation (Compose, ViewModel)  ─→ domain (UseCase, Repository interface)  ─→ data (Room, DataStore, Repository impl)
```

- **UI** не знает о Room. **Domain** не знает о Compose. Никогда не нарушай слои.
- **MVVM + UDF:** ViewModel хранит `StateFlow<UiState>`, UI отправляет события через функции.
- **Один UseCase = одна доменная операция.** Composable вызывает ViewModel, ViewModel вызывает UseCase.

## ⛔ Категорические запреты

- ❌ Никаких облачных сервисов, аналитики, рекламы.
- ❌ Никакого `runBlocking` в production-коде.
- ❌ Никакого `LiveData` — только `StateFlow`/`Flow`.
- ❌ Никаких хардкодов строк/цветов/размеров вне ресурсов и темы.
- ❌ Никаких новых permissions сверх перечисленных в README §2.
- ❌ Никаких SDK, требующих сети.

## ✅ Обязательные правила

- ✅ Каждый UseCase — с unit-тестом.
- ✅ Каждый нетривиальный Composable — с `@Preview` (светлая + тёмная).
- ✅ Все строки — в `res/values/strings.xml`.
- ✅ Все цвета — через `MaterialTheme.colorScheme.*`.
- ✅ Все размеры — через `dp`/`sp` константы или `MaterialTheme.spacing` (если введёшь).
- ✅ Каждый Composable, принимающий `Modifier`, ставит его первым необязательным параметром.

## 🔍 Чек-лист перед коммитом

1. Сборка чистая, без warnings.
2. `./gradlew lint detekt test` проходит.
3. Превью рендерятся.
4. Светлая и тёмная темы проверены.
5. Нет хардкодов.
6. Обновлён `docs/CHANGELOG.md`.

## 🤔 Когда задавать вопросы пользователю

**Спрашивай**, если:
- Требование в ТЗ можно интерпретировать двумя способами.
- Возникла техническая дилемма с долгосрочными последствиями (выбор библиотеки, паттерна).
- Нужно сделать выбор между двумя вариантами UX, не зафиксированный в ТЗ.

**Не спрашивай**, если:
- Ответ есть в `README.md` (перечитай).
- Это рутинное архитектурное решение, соответствующее правилам в этом файле.

## 📝 Стиль вопросов к пользователю

Когда задаёшь вопрос:
1. Опиши контекст в 1-2 строках.
2. Дай 2-3 варианта с pro/con.
3. Укажи свою рекомендацию.

**Плохо:** "Как обрабатывать эту ошибку?"  
**Хорошо:** "При попытке отметить привычку без сети — ситуация невозможна (БД локальная), но если БД повреждена — варианты:  
(a) показать `AimSnackbar` с retry,  
(b) показать `AimAlertDialog` с предложением restore из бэкапа,  
(c) краш с отчётом.  
Рекомендую (a) — наименее травматично. Подтверждаешь?"
