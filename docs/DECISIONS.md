## ADR-0002 — Шрифт Inter через Downloadable Fonts

**Дата:** 2026-05-21
**Статус:** принято
**Контекст:** README §7.3 требует Inter с весами Regular / Medium / SemiBold / Bold и поддержкой кириллицы. Альтернативы — bundled `.ttf` или Variable Font.
**Решение:** Downloadable Fonts через Google Fonts provider (`com.google.android.gms.fonts`). Зависимость `androidx.compose.ui:ui-text-google-fonts`. Сертификаты GMS — `res/values/font_certs.xml` (канонический Apache 2.0 файл из AOSP).
**Последствия:**
- APK не растёт от веса шрифтов.
- На первый запуск без интернета — fallback на системный san-serif.
- Требует наличие Google Play Services (приемлемо для Android 13+).

## ADR-0008 — Soft delete без каскада на data-слое

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** При удалении цели задачи внутри неё нужно прятать из активных запросов, но не дублировать каждую задачу в корзине. Альтернативы:
- (a) Cascade soft delete: при удалении цели проставлять `deletedAt` на все задачи с тем же timestamp. Минус: каждая задача отдельной строкой в корзине; восстановление требует сложной логики «по той же группе».
- (b) Без каскада: только цель помечается `deletedAt`. Активный запрос задач JOIN'ится с `goals` и фильтрует `WHERE g.deleted_at IS NULL`. Минус: запросы чуть сложнее.
**Решение:** (b). Каждая сущность ведёт свой собственный `deletedAt`. Видимость задач определяется JOIN'ом с целью. Восстановление цели автоматически восстанавливает все её задачи (их `deletedAt` остался NULL). Запрос на корзинные задачи: `WHERE t.deleted_at IS NOT NULL AND g.deleted_at IS NULL` (то есть в корзине показываем только индивидуально удалённые задачи живых целей).
**Последствия:** простой UX — клик «Восстановить» на цели возвращает всё содержимое одним движением. Индивидуально удалённые задачи в корзине отображаются как отдельные строки. Hard delete (через 30 дней или вручную) каскадно удаляет задачи через FK ON DELETE CASCADE.

## ADR-0009 — Recurrence как JSON-строка в колонке

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** `Recurrence` — sealed class с 5 вариантами (Daily, WeeklyOn(days), Weekly, EveryNDays(n), Monthly). Альтернативы хранения:
- (a) Отдельная нормализованная таблица `task_recurrences` с дискриминатором и колонками для всех вариантов.
- (b) JSON-строка в колонке `tasks.recurrence`.
- (c) Несколько колонок в `tasks` (recurrence_type, recurrence_days, recurrence_n).
**Решение:** (b). Recurrence читается всегда вместе с задачей, никогда не запрашивается отдельно. kotlinx.serialization уже подключён. Schema проще, миграции при добавлении нового варианта не нужны.
**Последствия:** невозможно фильтровать задачи по типу повторения через SQL (приходится в Kotlin). Для текущего MVP это не нужно. Если в будущем понадобится индекс по recurrence — мигрируем на (a).

## ADR-0010 — Drag-n-drop задач: только внутри уровня (Sprint 2)

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** README §6.2 описывает DnD задач, включая «перемещение между уровнями». Полная реализация требует значительной работы по UX (визуализация позиции drop'а в чужом поддереве, ограничения по depth, обработка циклов). Спринт 2 уже самый большой.
**Решение:** в Sprint 2 поддерживается только перетаскивание между братьями одного `parentTaskId`. В `onMove` callback библиотеки `sh.calvin.reorderable` отклоняем перемещение, если у source и target разные родители. Смена родителя в данных делается через `MoveTaskUseCase` (использует репозиторий с проверкой циклов и глубины) — UI-вход в Sprint 8 (Полировка).
**Последствия:** drag-n-drop работает для типичного сценария «переставить задачи в списке». Перенос между ветками дерева — пока через delete + create заново (грубо, но это можно). В Sprint 8 — full DnD с подсветкой возможных drop-зон.

## ADR-0013 — Hilt assisted injection для форм Edit*

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** `GoalEditBottomSheet` и `TaskEditBottomSheet` принимают `mode: Create | Edit(id)` в конструкторе ViewModel'а. Через стандартный `@HiltViewModel + hiltViewModel()` так нельзя — параметр нужен в момент создания.
**Решение:** `@HiltViewModel(assistedFactory = X.Factory::class)` + `@AssistedInject` + `@AssistedFactory` (поддержано в Hilt 2.49+). В Compose: `hiltViewModel<VM, VM.Factory>(key, creationCallback = { f -> f.create(mode) })`. Key включает mode-id, чтобы разные режимы (создание / редактирование разных целей) получали отдельные VM-экземпляры.
**Последствия:** чистая зависимость только на mode, без передачи `SavedStateHandle` руками. Альтернатива через Navigation arguments была бы громоздкой для bottom sheet'ов, которые не являются destination'ами.

## ADR-0014 — Цикл отметок привычки: нет → DONE → FAILED → нет

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** README §6.3 описывает 3 состояния отметки (DONE / FAILED / нет). Нужно UX-решение «как менять отметку тапом по календарю». Альтернативы:
- (a) Тап → popup с тремя кнопками.
- (b) Циклический сдвиг: нет → DONE → FAILED → нет.
- (c) Долгий тап для FAILED.
**Решение:** (b). Простой одиночный тап в типичном кейсе сразу даёт DONE; повторный тап превращает в FAILED (отметка «сорвался»); третий тап убирает отметку. Без popup'ов, минимум кликов для типичного сценария.
**Последствия:** случайно сделанный FAILED легко отменить ещё одним тапом. Пользователю нужно один раз увидеть pattern, чтобы понять — UX-полировка ониборда в Sprint 8.

## ADR-0015 — Стрик в TimesPerWeek: пустая неделя считается успешной?

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** README §6.3: «Прерывается, если в неделе ≥1 `FAILED` И <n `DONE`». Это формальное правило не определяет, что делать с неделями без любых отметок (done=0, failed=0).
**Решение:** для **best** streak — пустая прошлая неделя ломает streak (это разумно: пользователь забил на привычку → стрик обнуляется). Для **current** streak — пустая прошлая неделя тоже ломает (нельзя считать, что бесконечные пропущенные недели входят в стрик). Текущая неделя (in-progress) — продолжает streak, даже если ещё пустая, чтобы пользователь не видел «0» в среду до первой отметки.
**Последствия:** небольшое отклонение от буквального толкования README — но удобнее UX. Если пользователь возразит — править одну строчку в `CalculateStreakUseCase.weeklyStreak`.

## ADR-0017 — Связь привычки с целью: `ON DELETE SET NULL`

**Дата:** 2026-05-22
**Статус:** принято
**Контекст:** При перманентном удалении цели — что делать с привычками, которые на неё ссылались? Варианты: каскадно удалить, оставить с null, оставить со старым ID-указателем.
**Решение:** `ON DELETE SET NULL` на FK `habits.goal_id`. Привычка сохраняется, теряет ссылку на цель.
**Последствия:** пользователь не теряет годами накопленные стрики при перепланировании целей. Soft delete цели на привычки не влияет (FK срабатывает только при hard delete).

## ADR-0018 — «Сегодня»: меню вместо swipe/DnD в Sprint 4

**Дата:** 2026-05-23
**Статус:** принято
**Контекст:** README §6.4 описывает swipe-жесты (вправо — выполнить, влево — меню) и drag-n-drop ручной сортировки на экране «Сегодня». Жесты невозможно проверить автотестами, а реализуются они нетривиально (конфликт SwipeToDismissBox с reorder, edge-cases отмены).
**Решение:** в Sprint 4 действия экрана «Сегодня» реализованы через явный чекбокс (тап = выполнить, с пружинной анимацией) + меню «⋮» (snooze на завтра / +3 дня / изменить дату / удалить; для привычек — «Сорвался»). Swipe и drag-n-drop переносятся в Sprint 8 (Полировка), где идёт работа над жестами и анимациями централизованно.
**Последствия:** вся функциональность доступна и надёжна без жестов. В Sprint 8 добавим swipe поверх (не ломая меню) и ручную сортировку.

## ADR-0019 — Lazy-материализация occurrences регулярных задач

**Дата:** 2026-05-23
**Статус:** принято
**Контекст:** README §6.2 предлагает два пути для регулярных задач: генерировать экземпляры на N дней вперёд (eager) либо материализовать лениво. Таблица `task_occurrences` нужна в любом случае.
**Решение:** ленивый путь. Строка в `task_occurrences` создаётся ТОЛЬКО при отметке выполнения экземпляра на дату. Отсутствие строки = «не выполнено». Появление задачи в «Сегодня» вычисляется на лету через `RecurrenceMatcher` (без записи в БД).
**Последствия:** нет фонового воркера генерации, нет «протухших» будущих строк, минимум записей. Минус: статистика по регулярным задачам считается по факту отметок (для Sprint 6 этого достаточно). Поле `status` в таблице оставлено для будущих состояний (SKIPPED и т.п.), сейчас всегда COMPLETED.
